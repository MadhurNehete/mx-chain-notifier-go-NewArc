Executive Summary
A focused security audit of the mx-chain-notifier-go codebase was conducted after
the remediation of the initial five findings (NOTIFIER-1 through NOTIFIER-5). This
follow-up review identified two new, genuine security vulnerabilities that are
remotely exploitable and represent real threats to the confidentiality and availability
of the service. Both findings are centered on the unauthenticated WebSocket
endpoint ( /hub/ws ), which lacks both origin validation and connection-rate controls.

1. Cross-Site WebSocket Hijacking via Open
Origin Check (CWE-346)
File: dispatcher/ws/wsUpgrader.go · Line: 27
Domain: WebSocket / Security · Origin: Introduced
Severity: High - CWE-346 (Origin Validation Error)
Decision: REJECT

What This Error Is
The WebSocket upgrader uses a CheckOrigin callback that unconditionally returns
true , completely disabling the browser's Same-Origin Policy protection for

WebSocket connections. Combined with the fact that the /hub/ws endpoint has no
authentication configured in api.toml , this allows a Cross-Site WebSocket
Hijacking (CSWSH) attack. An attacker can host a malicious webpage that opens a

Page 2 of 11

WebSocket connection to the notifier from any origin, receiving the full stream of
blockchain events and subscribing to targeted address activity — all without any
authorization.

Attack Scenario
1. Attacker hosts a malicious page at evil.com .
2. A legitimate user with network access to the notifier visits evil.com .
3. The malicious page's JavaScript opens a WebSocket to ws://<notifierhost>:5000/hub/ws .

4. The connection is accepted because CheckOrigin returns true regardless
of origin.
5. The attacker receives all blockchain events (transactions, SCRs, state
accesses).
6. The attacker sends subscription messages to monitor specific addresses in
real-time.

The fix replaces the blanket return true with an origin-aware validation. When a
browser initiates a WebSocket handshake, it automatically includes an Origin
header indicating the page's domain. The corrected CheckOrigin function inspects
this header: if it is empty (as is the case for non-browser clients such as Go
WebSocket clients or wscat ), the connection is allowed through, preserving

Page 4 of 11

backward compatibility with existing observer integrations. If an Origin header is
present, the function returns false , rejecting the cross-origin browser request. This
prevents a malicious webpage from hijacking the WebSocket connection while
leaving programmatic (server-to-server) clients unaffected.

[!NOTE]
For deployments that require browser-based WebSocket access, extend the
CheckOrigin function to validate against a configurable allowlist of trusted

origins. Additionally, consider adding Auth = true to the /ws route in
api.toml to require authentication for all WebSocket connections.

2. Denial of Service via Unbounded WebSocket
Connections (CWE-770)
File: dispatcher/ws/wsHandler.go · Line: 54-74
Domain: WebSocket / Availability · Origin: Introduced
Severity: High - CWE-770 (Allocation of Resources Without Limits or Throttling)
Decision: REJECT

What This Error Is
The WebSocket handler ( ServeHTTP ) accepts every incoming connection and
spawns two persistent goroutines per connection ( readPump and writePump ), each
with an associated 256-slot buffered channel. There is no limit on the total number of
concurrent WebSocket connections. Since the /hub/ws endpoint is
unauthenticated, an attacker can open tens of thousands of connections, exhausting
the server's memory and file descriptors, causing a complete denial of service.

Page 5 of 11

Attack Scenario
1. Attacker writes a simple script that opens WebSocket connections in a loop to
ws://<notifier-host>:5000/hub/ws .

2. Each connection is accepted unconditionally (no auth, no origin check, no cap).
3. Each connection consumes: 2 goroutines (~16KB stack each) + 256-slot send
buffer.
4. At 10,000 connections: ~160MB goroutine stacks + channel overhead + file
descriptors.
5. At 50,000+ connections: OS file descriptor limit exhausted or OOM kill triggered.
6. Legitimate observer and subscriber connections are refused or existing ones
are disrupted.

The fix introduces an atomic.Int64 counter ( connCount ) to the
websocketProcessor struct that tracks the number of active WebSocket

connections. Before upgrading any incoming HTTP request to a WebSocket, the
handler checks whether the current count has reached the maxWSConnections
ceiling (1024). If the limit is reached, the request is rejected immediately with an
HTTP 503 (Service Unavailable) response, preventing the connection from being
established and no goroutines are spawned. When a connection is successfully
established, the counter is atomically incremented. The writePump goroutine is
wrapped in an anonymous function that decrements the counter when the pump
exits (i.e., when the connection closes), ensuring the count always reflects reality.
This bounds the total resource consumption to a predictable maximum regardless of
how many connection attempts an attacker makes.

The maxWSConnections constant should be made configurable via
config.toml . Additionally, consider implementing per-IP connection limits to

prevent a single attacker from consuming the entire connection pool.

3. Unauthenticated Event Injection on Mutating
Endpoints (CWE-306)
Severity: High - CWE-306 (Missing Authentication for Critical Function)
Status: FIXED

What This Error Is
The core mutating endpoints of the Notifier ( /events/push , /events/revert , and
/events/finalized ) were initially configured with Open = true and Auth =
false in the default api.toml . These endpoints allow external actors to inject

blockchain events, trigger reverts, and finalize blocks. Without authentication, any
network-adjacent attacker could send malicious payloads to these endpoints, leading
to state corruption in downstream dApps or denial of service by flooding the event
stream with fake data.

Fix
The remediation involved two layers: 1. Enforced Authentication: Changed Auth =
true for all mutating routes in api.toml . 2. Payload Sizing: Implemented
http.MaxBytesReader in the eventsGroup to limit request bodies to 10MB,

preventing memory exhaustion from large malicious payloads.
# cmd/notifier/config/api.toml (FIXED)
[APIPackages.events]
Routes = [
{ Name = "/push", Open = true, Auth = true },
{ Name = "/revert", Open = true, Auth = true },
{ Name = "/finalized", Open = true, Auth = true },
]

Vulnerability Remediation: Side Effects and
Implementation Notes
The architectural changes required to secure the Notifier introduced specific side
effects that necessitated updates to the integration test suite. These are documented
here to assist in future maintenance.

1. Asynchronous Publishing & Test Latency
Security Change: Transitioned the event publisher to an asynchronous model using
bounded channels (size 1024) and a background processing loop. This prevents a
slow consumer (e.g., a stalled RabbitMQ instance) from blocking the main API
threads, which is a critical protection against Denial of Service.
Side Effect: Integration tests that perform an action (e.g., POST /events/push ) and
immediately assert a result (e.g., check Redis for the entry) began to flake or fail. This
is because the API returns a successful response as soon as the event is queued, but
the background processing might take a few milliseconds to complete the write.
Remediation: Added explicit time.Sleep and increased WaitTimeout durations in
integrationTests/rabbitmq and integrationTests/websocket to allow the

background publisher threads to synchronize before assertions are made.

2. Publisher Timeout Calibration
Security Change: Implemented a publisherSendTimeout (set to 100ms) to ensure
that the Notifier never deadlocks if the internal message queue is full.
Side Effect: Unit tests that flood the publisher (e.g.,
TestPublisher_BroadcastDoesNotBlockIndefinitelyWhenQueueIsFull )

required adjustment to their internal watchdog timers to accommodate the expected
aggregate delay of 100ms per overflow message.

3. Origin Validation Fallback
Security Change: Implemented secure CheckOrigin in the WebSocket upgrader.

Page 10 of 11

Side Effect: The Go-based WebSocket clients used in integration tests sometimes
send an Origin header that doesn't match the host or is unexpected.
Remediation: The CheckOrigin logic was refined to use a same-host host matching
logic ( u.Host == r.Host ) and a fallback to allow empty origins (common for nonbrowser programmatic clients). This maintains high security for browser-based
clients while ensuring integration tests and server-to-server communication remain
functional.

Final Audit Conclusion: All identified critical and high-severity issues have been
successfully remediated. The system is now hardened against CSWSH, WebSocketbased DoS, and unauthenticated data injection. All integration and unit tests have
been updated to align with the new secure architecture and are passing.
Merge Recommendation: APPROVE (Post-remediation)
